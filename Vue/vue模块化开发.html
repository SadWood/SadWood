# 1、什么是模块化

  • 将项目 拆分成 多个单元（组件），使用单元（组件）组合的方式，构建项目整体
  • 优点：组件或者功能模块 可以重复使用，减少相同代码的重复定义次数
  • 概念上看，简化项目的开发，实际上模块化开发从结构到代码上，并不会出现简化代码的功能，反而会让项目结构复杂化
    • 不是在开发阶段简化代码
    • 为了在后期项目维护和更新时简化操作量
    • 缺点：结构复杂开发繁琐
    • 提供了良好的代码管理结构

# 4.1、axios

  • axios 不支持 JSONP 请求的
  • 在模块化项目中 如何使用
npm | cnpm install axios -S
  • 项目中如何使用
axios.request(config)         ==>  $.ajax()

axios.get(url[, config])        ==>  $.get()
axios.delete(url[, config])
axios.head(url[, config])
axios.options(url[, config])
axios.post(url[, data[, config]])    ==> $.post()
axios.put(url[, data[, config]])
axios.patch(url[, data[, config]])
  • HTTP 请求协议规范中 ，get post put delete patch head ……
    • get 用于获取数据
    • post 用于填加数据
    • put 用于修改数据
    • delete 用于删除数据
  • Restful 风格请求 由前后端配合完成的一种 请求分类技术
    • 区别与传统 http 请求，以请求协议不同 区分用户得操作
  • vue 中的使用

import axios from "axios";

export default {
    data(){

            bookList:[]
        }
    },

        getBooks(){

            // let p = axios.get("static/datas/books.json");
            let p = axios.get("http://127.0.0.1:80/bookApi/book.php");
            // p.then(function(response){
            p.then((response)=>{
                // response ajax 请求 响应对象返回
                //   包含请求状态，请求结果，请求响应码……
                //   response.data 才是返回的后台数据
               
                // try {
                //     console.log(response);
                //     console.log(response.data);
                // } catch (error) {
                //     return new Error("请求失败")
                // }

                if(response.statusText!="OK"||!response.data.resultState){
                    // return new Error("请求失败");
                    /*
                     * throw 关键和 return 具有一个相同的功能，结束当前代码运行
                     *      return 在数据层面中 表示 返回值
                     *      throw 表示抛出错误或者异常 
                     */
                    console.log(1);
                    throw new Error("请求失败");
                }
                // console.log(response.data);
                this.bookList = response.data.result;

            }).catch(function(error){

                // alert(error.message);

proxyTable: {

# 5.2.2、路由地址的 特殊参数

  • / 路由的默认组件访问地址 ==> 默认访问组件的定义
  • * 路由地址统配符 ==> 完成 资源地址错误的 显示
    • 资源地址错误：路由组件方式时，所有在 路由表中 未被定义 其它地址
  • 路由表定义
const router = new Router({
  // key：描述路由的配置属性   value：属性对应的参数值
  // routes 定义 请求地址和组件的关系  Array[object,object]
  routes:[
    
    // 对象是为了 提供每个组件 对于 路由的描述
    // {path:"",component:"",redirect},
    //    path 自定义组件路径名称
    //    component  定义组件 关联 path 指向的组件是谁
    //    redirect  资源 重定向  ==>  重新 定位 到 需要资源上
    {path:"/",redirect:"/home"}, // 默认首页的定义
    {path:"/home",component:HomePage},
    {path:"/aboutMe",component:AboutMePage},


    {path:"*",component:NotFound}
  ]
});

# 5.2.3、默认的路由激活样式

读取 URL 中的 路由地址的指向 ，和 标签中 <router-link> 所定义的 跳转地址进行比较，如果相同 装载样式

  • router-link-exact-active ： 地址完全匹配 会被装载的样式
  • router-link-active ： 地址包含匹配 层级装载的样式

# 5.3、路由的全局配置

  • linkActiveClass ： 重命名 router-link-active
  • linkExactActiveClass ： 重命名 router-link-exact-active
  • base : 完成项目基础路径定义 ==> 控制组件
  • mode : 路由的地址模式
    • "hash" ： 资源定位模式 ==> url 会存在明显的页面分割界限 /#
    • "history" : 路由的主动拦截模式 ==> 用于模拟 传统 URL地址的方式
      • 配置了 history 模式后，需要 在 index.html 通过 <base> 标签完成 静态资源路径的定义

import Vue from 'vue';

import HomePage from '@/pages/HomePage';
import AboutMePage from '@/pages/AboutMePage';

    mode:"history",
  
    // key：描述路由的配置属性   value：属性对应的参数值
    // routes 定义 请求地址和组件的关系  Array[object,object]
    routes:[
      
      // 对象是为了 提供每个组件 对于 路由的描述
      // {path:"",component:"",redirect},

      {path:"/aboutMe",component:AboutMePage},

    ]

# 5.6.1、路由的基础定义

  • 需要在定义路由时 ，完成 路由层级的 关联关系
  • 在 routes 属性中 直接定义的 路由 ==> 一级路由|根路由
  • 所有在根路由中 定义路由 ==> 子路由 | 二级路由
    • 所有二级路由中定义路由 ==> 三级路由
      • ……

# 5.6.2、路由的寻址机制

  • 路由的地址寻址==> 以 URL（组件路由地址的） 地址的层级，来匹配 页面中 router-view的层级
  • 项目路由树

# 5.6.3、嵌套路由对激活样式的影响

读取 URL 中的 路由地址的指向 ，和 标签中 <router-link> 所定义的 跳转地址进行比较，如果相同 装载样式

  • router-link-exact-active ： 在路由跳转时 URL 的 组件地址链 和 标签中定义的地址完全匹配 会被装载的样式
  • router-link-active ： 在路由跳转时 URL 的 组件地址链中 包含 标签中定义的 地址 层级装载的样式

# 5.7.2、reset 风格参数传递

  • 在路由地址后 定义参数 （传统的 URL 地址传参数）

    • URL 本身的设计 只是为了针对 页面路径显示使用，URL的地址传参==> 可以传递的数据大小是有限

  • reset 风格参数设计

    • 基于 URL地址的方式实现参数的共享

    • 参数传递时，省略需要传递的参数的KEY （属性名）

    • 在地址中 以 / 分割多个参数

    • 产生问题：导致 代码 无法 主动分析 URL中那部分是 地址 ，那部分是 参数

      • 在定义 路由指向时，通过 : 在地址后 定义 传递参数的 名称

      • 例子：http://127.0.0.1:8080/testReset/参数1/参数2……

      • vue路由定义：{path:"/testReset/:aa/:bb/……"}

    • reset风格参数和get 风格参数 ，本身在获取时 需要 参数独立

        mounted() {
        // query 属性 专用于 get 参数的获取
        console.log(this.$route.query)
        // params 属性 专用于 post 参数的获取
        console.log(this.$route.params)
        this.loadBookInfo(this.$route.params.bookId);
        this.loadComment(this.$route.params.bookId);
    }

    • Vue 为了简化 开发者 对于 reset 参数的获取难度，可以自动解析

      • 在配置路由时 使用 props:true 方式开启 参数的自动获取

      • 在目标组件中 以 props 的方式接收参数

      • 路由配置

        {path:":bookId/bookDetail",component:BookDetail,props:true}

      • 参数获取

        export default {
    props:{
        //参数被自动装载
        bookId:""
    },
    mounted() {
        this.loadBookInfo(this.bookId);
        this.loadComment(this.bookId);
    },
};
